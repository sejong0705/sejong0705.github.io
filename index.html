주요 코드

  • Firebase Firestore를 활용하여 NoSQL 형식으로 데이터 구조 설계 및 관리
  • 비동기 처리 (async/await)를 사용하여 데이터 로딩 이후 UI를 즉시 반영

문제 해결 경험

🔎 문제
처음 접하는 NoSQL (Firebase Firestore) 환경에서, 기존 관계형 데이터베이스(RDB)의 테이블 기반 설계 방식과는 다른 문서(Document) 기반 데이터 구조 설계에 어려움을 겪었습니다.
🛠 해결 과정
반복적인 데이터 모델링과 구조 테스트를 통해 루틴, 통계, 기록 데이터를 효과적으로 저장할 수 있는 구조를 찾아갔습니다.
Firebase의 서브컬렉션, 중첩 필드 등 문서형 DB의 특성을 직접 적용하며 설계를 점차 개선해 나갔습니다.
✅ 결과
NoSQL의 유연성과 구조적 장점을 실무에 적용해보며, 데이터 흐름을 고려한 설계 역량이 향상되었습니다.
실제 앱 동작 흐름에 맞춘 구조를 구성할 수 있게 되었고, 유지보수와 확장에 용이한 데이터 구조를 완성할 수 있었습니다.

문제 해결 경험

🔎 문제
프로젝트 초기에는 유저 기능이 미완성된 상태여서, 기능 테스트를 위해 유저 ID를 직접 하드코딩해 개발을 진행했습니다.
하지만 계정마다 동작이 다르게 나타나는 기능에서는 혼란과 테스트 한계가 발생했습니다.
🛠 해결 과정
팀원과 협의해 최소한의 User 엔티티와 로그인 연동 구조를 도입하였고, 여러 개의 더미 계정을 생성해 실제 사용 흐름에 가까운 테스트 환경을 구축했습니다.
또한 Spring Security에서 인증된 사용자 정보를 가져오는 CurrentUserProvider를 적용하여 코드 전반을 개선했습니다.
✅ 결과
실제 유저 기반의 흐름으로 기능 테스트가 가능해졌고, 예외 처리 및 접근 제어 로직도 명확히 구현할 수 있었습니다.
그 과정에서 Spring Security 인증 처리 방식에 대한 이해도 깊어졌습니다.
